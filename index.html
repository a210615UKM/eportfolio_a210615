Coursework

Lab Submissions

Five hands-on labs covering the foundations of modern Android development with Jetpack Compose.

Lab 01

UI with Jetpack Compose

Basic Layout · Composables · Modifiers
Reflection: My first encounter with Jetpack Compose felt different from XML — declarative, predictable, and fast to iterate. I learned how Column, Row, and Modifier compose into expressive UIs. Building the layout taught me the importance of spacing, alignment, and reusable composables for clean code.
▶ Watch Demo YouTube ↗
Lab 02

Interaction in Compose

State · Events · remember & mutableStateOf
Reflection: This lab introduced me to state hoisting and the unidirectional data flow that makes Compose so powerful. I learned to handle onClick events, manage state with remember, and watch the UI react instantly to changes. It clicked when I realised the UI is just a function of state.
▶ Watch Demo YouTube ↗
Lab 03

Material Design

Theming · Typography · Components
Reflection: Designing with Material 3 taught me that consistency and accessibility aren't extras — they're the baseline. I explored color schemes, typography scales, and components like Card, Scaffold, and TopAppBar. Good design speaks before a single word is read.
▶ Watch Demo YouTube ↗
Lab 04

Navigation & ViewModel

NavHost · Routes · MVVM
Reflection: Multi-screen apps finally made sense once I understood NavHost and NavController. Pairing navigation with ViewModel kept the UI stateless and survivable across config changes. This was the moment my apps started to feel like real apps.
▶ Watch Demo ⌥ GitHub Repo
Lab 05

Room — Local Database

Entities · DAO · Coroutines · Flow
Reflection: Room turned my apps from session-only to truly persistent. Setting up @Entity, @Dao, and the database, then querying with Flow taught me how reactive data layers feed Compose perfectly. Debugging migrations was tough, but rewarding.
▶ Watch Demo ⌥ GitHub Repo
Capstone Work

Project Submissions

Two capstone projects that bring together everything I learned — from UI fundamentals to API, sensors, and cloud integration.

Project 01 · Built on Lab 4

Humanity — Foundation Build

Problem statement: Many willing donors and volunteers don't know where to give, how to find verified causes, or how to access poverty-reducing job opportunities. Existing platforms scatter these touchpoints across multiple apps.

What I built:

  • Multi-screen Compose app with NavHost navigation
  • Donation, jobs, and volunteer flows in one platform
  • MVVM with ViewModel for clean state management
  • Material 3 components for a consistent look and feel

Lessons learned: Architecture matters early. Separating screens, view models, and data sources from day one paid off. I also learned that good UX means cutting features, not adding them.

▶ Watch VSR ⌥ Project 1 Repo Lab 4 Repo ↗
Project 02 · Final Capstone

Humanity — Full Platform

Problem statement: Poverty in Malaysia is fueled by unemployment, disaster, and limited access to support — yet help is fragmented. Humanity unites donations, jobs, and volunteer activities so anyone can take meaningful action with one tap.

Features delivered:

  • 2 core screens — Donation flow + Job & volunteer hub
  • API integration for live charity & opportunity data
  • Sensors for location-aware volunteer matching
  • Cloud backend for storage and authentication
  • Room for offline-first reliability

Lessons learned: Real apps are integration problems. Wiring up an API, sensors, and cloud taught me to handle errors gracefully, design for offline, and write code that survives bad networks. Ship small, ship often, and listen to the user.

Looking Back

Reflection & Learning Journey

From Pixel-Pushing to State-Thinking

Lab 1 was about painting pixels with composables. By Lab 2 I was thinking in state. That shift — from "how do I draw it" to "what is the state of this UI" — was the single biggest mindset change of the semester.

Design as a First-Class Citizen

Lab 3 taught me that great Android apps follow Material guidelines not because they have to, but because users expect it. I stopped fighting the system and started leaning into it.

Architecting for Scale

Lab 4 forced me to separate UI, logic, and navigation. I refactored Project 1 three times before it felt right — and that struggle is exactly where I grew the most.

Persistence Changes Everything

Lab 5's Room database turned Humanity from a demo into something that actually remembers users. Combined with Flow, the data layer felt alive.

Challenges I Faced

Gradle errors at midnight, API responses that changed shape, sensor permissions on different Android versions, and the eternal struggle of pushing big folders to GitHub. Every bug taught me to read errors instead of fearing them.

SDG Impact

Building Humanity for SDG 1 reminded me why I chose Software Engineering. Code is leverage. A simple app can connect a donor to a flood victim, a job-seeker to an employer, or a volunteer to a community in need. That's the kind of impact I want to keep building.
